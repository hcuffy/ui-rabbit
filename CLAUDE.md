# CLAUDE.md — UI-Rabbit

> Read `UI-Rabbit-Build-Plan.md` (full stack, infra, deliverables D1–D7, agent prompts, milestones) before acting. This file is the quick brief; the build plan is the source of truth.

## What this is
Open-source AI agent: takes a plain-language charter prompt, drives a web app's UI via **Playwright**, finds issues, and **remembers** prior runs so each run suppresses known noise and surfaces only what changed. "CodeRabbit for UI." First target: **rabbit**.

## Settled decisions — DO NOT relitigate
Decided. Don't reopen unless a *new constraint* genuinely breaks one — then say so explicitly, don't silently work around it.
- **Language:** all-TypeScript/Node, front to back. No Python in the product. (The engine is built **natively in TS** — the earlier "port from a proven Python M1" plan is void: that Python engine was never actually written. See Current state.)
- **Driver:** Playwright (Node binding). **Not Cypress** — dropped (command-queue fights LLM nav; Playwright is a11y-native, first-class console/network signals, cheap context isolation, no language bridge).
- **Perception:** `page.ariaSnapshot({ boxes: true })` — YAML a11y tree + per-element boxes. Not raw HTML; not legacy `page.accessibility.snapshot()`.
- **Backend:** Fastify + **Zod** (single source of truth for `Finding`/`Run`/`AppMap`/`Baseline`, shared with frontend — no codegen). Mongo **native Node driver**.
- **Memory store:** MongoDB 8.x.
- **Judge:** one LLM call via `@anthropic-ai/sdk` tool-use; default `claude-sonnet-4-6`, escalate to `claude-opus-4-8` only on low confidence; gated behind deterministic oracle + baseline compare.
- **License:** MIT (decided at D1).
- **Explorer:** charter-scripted in MVP — **LLM is the judge, not the navigator.** LLM-driven exploration is post-MVP.
- **Auth:** dev env has no login → auth deferred; leave a `storageState` seam stubbed.
- **Scope now:** thin vertical slice M2 → M5 + the M1→TS port + a pulled-forward safety floor.
- **Still open (flag, don't assume):** whether to also emit Cypress repro specs for the 85% rabbit suite.

## How to work
- **Don't guess.** Unsure or research inconclusive → say so, let me decide. Propose an alternative only when you *know* it's better.
- **Review before presenting.** Write → review → audit → re-check. Catch stale APIs/config mismatches before I see them.
- **Verify current versions/APIs by searching** when it matters — this stack (esp. Playwright) moves fast.
- **Vertical slices, runnable code.** Create real files, run them, show output. Prove each step.
- **Cheapest oracle first.** Never put an LLM call where a deterministic check works. Call out cost implications.
- **Preserve the `Finding` contract** (shared Zod schema) across driver and judge swap-points.
- **Surface non-determinism early** — timestamps, random ids, animations, A/B flags, 9-language localization → mask before fingerprinting or they become false regressions.
- **Node hygiene:** `await` all IO, no floating promises, don't block the event loop, strict TS, no `any`.
- **Honest tradeoffs.** Name Mongo's wrong-fit risks, false-positive risk, Playwright limits. Don't oversell.
- **Don't redo finished work** (M0/M1 done). Flag if you'd skip ahead.

## Highest-risk surface
**State fingerprinting / stable screen identity** (normalized URL + heading anchor, volatile data masked) — harder and more important than the AI part. There is **no Python reference and no parity oracle** (the M1 engine was never built), so this gets built fresh in TS and de-risked with **hand-authored golden fixtures + TDD**: define expected normalized outputs by hand for a spread of inputs (volatile data, locales, ids, timestamps), assert against them. Get this piece exactly right; it carries no safety net from prior work.

## Current state
M0 done (planning/spec). **M1 was never actually implemented** — no `memory.py`/`oracle.py`/`runner.py`/`judge.py`/`demo.py` exist; the "proven Python engine" was planning-optimism. **D1–D7 complete — MVP feature-complete** (modulo the deferred validations noted below: judge quality eval, judge→UI end-to-end, live-rabbit wiring). D1: repo + infra skeleton, CI green, MIT. D2: engine built natively in TS (`packages/engine`: ariaTree, mask, screenId, fingerprint, oracle, dedup, runner, judge), all §6 golden fixtures pass. D3: Playwright driver (`packages/driver`) against a **local mock** (no live rabbit this session) — charter-scripted locations list+detail flow, real-chromium capture feeding the unmodified engine, a repro-spec generator, a CLI/test-only JSON local store (not Mongo), `storageState` auth seam stubbed. Mock target uses `context.route()` fulfillment against a fake origin (`http://mock.local`), not a real static server (driver-spec §2 allows either). The §8 three-run slice (run 1 learns → run 2 volatile-only suppresses → run 2′ changed-regression flags + emits a real `.spec.ts`) is proven both as an automated e2e test (`sliceProof.e2e.test.ts`) and manually via the CLI. Additive `shared` schema tweaks made at D2 (flagged): `STATE_DIVERGENCE` Finding type, `dedupKey` field, `networkErrors[].method`.

**D4** (`packages/backend`, per `docs/backend-spec.md`): Fastify + native Mongo driver repos (`RunRepo`/`FindingRepo`/`BaselineRepo`/`AppMapRepo`, one per memory layer, Zod-parse on every read/write, `_id` = the schema's own uuid/`screenId`) wrap the unmodified D2 engine + D3 driver. Orchestrator (`orchestrator.ts`) runs the loop in-process as a fire-and-forget background job that catches all its own errors and writes `FAILED` + `error` rather than ever rejecting (so an unhandled run error can't crash Fastify). Endpoints: `POST /runs` (202, returns immediately), `GET /runs`, `GET /runs/:id`, `GET /runs/:id/findings`, `GET /findings/:id`, `GET /findings/:id/repro`. Budget counters (`stepsUsed` etc.) are populated, not enforced (enforcement is D7, per spec §8). Additive `shared` schema tweak made at D4 (flagged, same pattern as D2's): `Finding.reproSpecPath` (optional) — backend-spec §4.6/§9.3 proposed "write repro `.spec.ts` to disk, store the path on the finding"; there was no existing field for it, so one was added. **Restart-survival proven manually** against the docker-compose Mongo: a throwaway HTTP mock server (`packages/backend/scripts/manual-mock-server.ts`, D4-proof-only, not product code) stood in for live rabbit access (still not available this session); run 1 against it learned 2 baselines + an `AppMap`, the backend process was killed and a *new* process started against the same Mongo, run 2 (same target) suppressed with zero `STATE_DIVERGENCE` findings (reading run 1's baselines back from a fresh process — memory survived the restart), and a third run after switching the mock to its `changed-regression` variant flagged a `NEW STATE_DIVERGENCE` and wrote a real repro spec to disk.

**D5** (`packages/engine/src/judge.ts`, per `docs/judge-spec.md`) — **code-complete and hardened.** `mockJudge()` replaced with a real `@anthropic-ai/sdk` tool-use call (`runJudge`) — one forced-tool call (`submit_verdict`, input validated with Zod before trusting it) per `STATE_DIVERGENCE` draft, default `claude-sonnet-4-6`, escalating to `claude-opus-4-8` only when Sonnet's confidence is `< 0.5`; still-low Opus confidence degrades to `NEEDS_HUMAN` rather than a forced guess. The SDK client is injected as a **lazy factory** (`JudgeRunOptions.clientFactory: () => AnthropicLike`, not a pre-built client) — `new Anthropic()` only runs on the first real divergence in a run, so a clean run and the driver mock CLI demo work with **no `ANTHROPIC_API_KEY` set at all**; `cli.ts`/`server.ts` memoize the real client behind that factory. Malformed tool output, a refusal (empty `content`), and a thrown SDK error (network/auth/rate-limit) all degrade to `NEEDS_HUMAN` — never a crash — but the two are now **tagged distinctly**: a thrown error is logged loudly via `console.error` and carries `reasoning: "Judge unavailable (infra/auth error): <message>"`, so a missing/invalid key reads as an infra failure, never as the model being cautious; a clean-but-unusable response keeps the plain `"Judge returned no parseable verdict."` wording. This rippled into `runEngineLoop`, which is now **async** (the judge call is a real network round-trip) and takes two new `EngineLoopInput` fields: `charter` (threaded to the judge as context) and `judge: JudgeRunOptions`; `maxLlmCalls`/`maxUsdPerRun` are optional per-run budget caps enforced by a `BudgetTracker` inside the loop — once hit, remaining divergences in that run skip the judge and become `NEEDS_HUMAN` rather than stopping the run. `EngineLoopOutput` gained `llmCallsUsed`/`costUsd`, which the backend orchestrator copies straight into the `Run` document (previously always 0). The repro-spec gate (driver `cli.ts` and backend `orchestrator.ts`) flipped from the D3/D4 placeholder (`STATE_DIVERGENCE && status NEW`) to the spec'd `verdict === "REGRESSION"`, now that real verdicts exist. Cost accounting uses a small cached $/MTok table for the two pinned models only; an overridden model string not in that table costs $0 rather than guessing. Tests: `judge.test.ts` covers confident-Sonnet, escalation-to-confident-Opus, escalation-to-still-low (`NEEDS_HUMAN`), malformed input, refusal, a thrown network error, a simulated invalid-API-key error, and an infra failure on the Opus escalation leg — each asserting the distinct infra-vs-confidence reasoning/logging — all via a mocked `AnthropicLike`, no real API in CI; `runner.test.ts` adds a cap-exceeded test (2 divergences, `maxLlmCalls: 1` → one judged, one `NEEDS_HUMAN`, `llmCallsUsed` stays `1`). 78 tests green repo-wide; build + lint clean. **UNVERIFIED — flag, don't assume covered:** the judge-spec §8 "manual sample eval" done-criterion (run the real judge against real divergences, confirm a human agrees with the verdicts) needs a real `ANTHROPIC_API_KEY` and **has not been run** — everything above is verified against a mocked client only. Run it before treating D5's judge quality (not just its plumbing) as proven. A standalone eval script for this exists: `packages/engine/scripts/judge-eval.ts` (dev tooling only, run via `ANTHROPIC_API_KEY=... npx tsx packages/engine/scripts/judge-eval.ts`) — prints per-case verdict/severity/confidence/reasoning for one real Playwright-captured divergence plus 3 hand-written intended-change cases, for eyeballing.

**D6** (`packages/frontend`, per `docs/frontend-spec.md`): React 19 + Vite 8 + TanStack Query v5 dashboard. New-run form (charter + targetBaseUrl, client-validated against a local Zod schema mirroring backend's `CreateRunBodySchema` — not exported from `shared`, so this one 2-field shape is unavoidably duplicated, not a Run/Finding/AppMap/Baseline fork) → run history (`GET /runs`) → run detail with 2s terminal-aware polling (`refetchInterval: (query) => isTerminalStatus(...) ? false : 2000`, v5's `(query)` signature, `isPending`/`isError` v5 names) → findings feed (`FindingCard`: type/verdict/severity/confidence/reasoning/evidence-summary + a repro download link gated on `finding.reproSpecPath`). All six D4 endpoints already existed — no missing-endpoint flag needed. Imports `Run`/`Finding` directly from `@ui-rabbit/shared`, Zod-parses every response against `RunSchema`/`FindingSchema` at the fetch boundary (`apiClient.ts`) — catches a real wire-format gap doing it: those schemas declare timestamps as `z.date()`, correct for the backend's Mongo `Date` instances, but HTTP JSON carries ISO strings, so the client revives the known date fields before validating (a JSON-serialization gap, not frontend/backend drift). Minimal plain CSS, no UI kit. Tests: Vitest + Testing Library + a `vi.stubGlobal("fetch", ...)` mock (no MSW, no real backend in CI) — `apiClient.test.ts`, `NewRunForm.test.tsx`, `FindingCard.test.tsx`, and a **real** stop-on-terminal polling test (`RunDetail.test.tsx`, real timers over ~4.5s — fake timers fought RTL/Query's internal scheduling too much to be worth it) proving a RUNNING run polls and a COMPLETED one doesn't poll again. 90 tests green repo-wide.

Two **necessary deviations from "no backend changes," both flagged, neither business logic**: (1) added `@fastify/cors` to `app.ts` (D6 shipped it as `origin: true` — reflect any origin; without it the browser blocks every frontend→backend call outright, discovered while running the manual click-through proof, not theoretical. **Superseded at D7**: pinned to an explicit `CORS_ORIGIN` allowlist — see the D7 paragraph below; `origin: true` no longer ships). (2) Frontend needs Vite's `bundler` module resolution (JSX, no explicit `.js` extensions on relative imports) and isn't consumed by any other package, so it's a non-composite leaf dropped from the root `tsconfig.json`'s project-reference list — root `tsc -b`/`pnpm run typecheck` no longer touch it; it typechecks via its own `tsc -b && vite build`.

**Manual proof actually run** (not just claimed): docker-compose Mongo + real backend dev server + real Vite dev server + a real headless-Chromium click-through (Playwright, throwaway scripts, deleted after) — triggered a run from the UI, watched it appear in history, watched the live view poll and stop exactly on COMPLETED, confirmed zero findings rendered for the clean baseline run. The repro-download path needed a seeded Finding (real judge calls need a key this session doesn't have, same gap as D5) — with one seeded, the finding card rendered every field correctly and the download link fetched real `.spec.ts` content with the right content-type. Caught and fixed two real bugs doing this: the CORS gap above, and confirmed (separately, on a bad hand-written test UUID) that zod v4's `.uuid()` correctly enforces the variant nibble, not just hex-and-dashes shape.

**D7** (`packages/backend/src/safety.ts`, per `docs/safety-spec.md`) — **the safety floor, last MVP deliverable.** Three pure, IO-free guards — `assertAllowedUrl`, `assertNotProdUrl`, `assertNotDestructive` — each throwing a typed `SafetyViolation` (`guard: "ALLOWLIST" | "PROD_URL" | "DESTRUCTIVE_ACTION"`) on trip, never doing IO themselves. Single source of truth: the **driver never imports `safety.ts`** (it can't — `backend` already depends on `driver`, the reverse would cycle) — `explore.ts` just gained two optional hook params (`onBeforeNavigate`, `onBeforeAction`) it calls and lets throw; the orchestrator is the only thing that constructs closures over the real guards and injects them. Pre-run, the orchestrator checks `targetBaseUrl` against the allowlist + prod patterns *before* `chromium.launch()` (cheapest possible refusal — `stepsUsed` stays `0`); per-navigation, the same checks re-run for every URL the driver actually visits, including a `clickFirstLink` destination resolved from the link's `href` *before* the click (an in-app link can point off-domain even when `targetBaseUrl` was on the allowlist). The destructive-action guard fires before the click itself, gated to `button`/`link` roles, word-boundary matched (`DEFAULT_DESTRUCTIVE_PATTERNS`: delete/remove/pay/purchase/confirm/cancel/charge/submit order/place order) — rarely trips today since charter-scripted exploration only does `navigate`+`clickFirstLink`, but it's the foundation the post-MVP LLM-driven explorer needs to be born safe. **Trip behavior, as decided:** any `SafetyViolation` is just a thrown error the orchestrator's existing catch already turns into `Run.FAILED` + the violation's own message as `error` — no new code path. **Budgets stayed graceful, untouched per the spec's explicit instruction:** D5's LLM-cap behavior (stop judging → `NEEDS_HUMAN`) is unchanged; `MAX_STEPS` is a plain `explore()` truncation (`maxSteps` slices the charter's step list, no throw) — the run still `COMPLETED`s with whatever it captured. `ALLOWED_DOMAINS` parses to an explicit host allowlist that's **fail-closed** (unset/empty ⇒ refuses everything, not "allow all"); `PROD_URL_PATTERNS` is an explicit regex denylist, not the heuristic "lacks a dev marker" variant of the spec's [CONFIRM] proposal — deliberately chosen since a positive "looks like dev" check risks silently admitting prod far more than an explicit pattern list risks over-refusing. **CORS pinned**: D6's `origin: true` replaced with `CORS_ORIGIN` (env, comma-separated, defaults to `http://localhost:5173`) reflected via `AppDeps.corsOrigins`; an unconfigured/wrong origin gets no `Access-Control-Allow-Origin` header at all. Tests: `safety.test.ts` (18 pure-unit cases — exact-host pass, off-list/lookalike-substring/empty-allowlist all rejected, prod-pattern defense-in-depth, destructive word-boundary matches vs. "Paying"/"Removable" false-positive checks, role-gating); `orchestrator.test.ts` gained 3 **real-Chromium** integration cases (off-allowlist target refused pre-launch, a custom-served page with an off-domain link aborts mid-run, a page with a "Delete Location" link aborts on the action guard) — no mocks for the guard logic itself, real Playwright navigation against `context.route()`-served HTML; `app.test.ts` gained a CORS reflection pair. 113 tests green repo-wide; build + lint clean. **Manual proof actually run**: real docker-compose Mongo + real backend with `ALLOWED_DOMAINS=localhost:5055` — an off-allowlist `POST /runs` failed in 3ms with the exact guard message, an allowed target completed normally in the same process.

MVP is now feature-complete. **Deferred validations (not blockers, but real gaps — don't assume covered):** the D5 judge-quality manual sample eval (needs a real `ANTHROPIC_API_KEY`, still not available this session — script ready at `packages/engine/scripts/judge-eval.ts`); the D6 repro-download path was only proven against a seeded Finding, never a real judge-produced `REGRESSION` (same key gap); live-rabbit wiring (the engine-spec's "[CONFIRM at D3]" items — query handling, literal id shape, German relative-time masking — never got a real target to confirm against).
